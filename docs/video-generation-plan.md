## Thesis

Video is the first MindRouter modality whose unit of work outlives an HTTP request, so the architectural bet is: make the GPU worker itself asynchronous, and make the durable authored document — not the request — the primitive. A gateway-side runner claims persisted jobs, routes each shot through the EXISTING registry/scorer/capacity machinery, and talks to the worker only in sub-second submit/poll/fetch calls, so `_proxy_with_retry`'s 180s `asyncio.wait_for` is never in the video path and the LLM hot path is never modified. Coherence is bought at the gateway — FLUX.2 (already deployed, no new GPU) generates approved per-entity reference sheets and keyframes, every shot carries the same reference set and seed policy, and hard cuts are the default — because no open model produces a coherent 30-60s clip in one call.

## Model stack

**Primary: Lightricks LTX-2.3 22B, served from `github.com/Lightricks/LTX-2`'s first-party `ltx_pipelines` package. No ComfyUI, no diffusers.**

Checkpoints (HF `Lightricks/LTX-2.3` + `Lightricks/LTX-2.3-fp8`):
- `ltx-2.3-22b-distilled-1.1.safetensors` — the v1 default. Two-stage 8+4 step (NOT a flat 8-step; verified from `utils/constants.py`: `DISTILLED_SIGMA_VALUES` is 9 values = 8 steps, plus a separate 4-value stage-2 schedule). Serves the `draft` and `standard` tiers.
- `ltx-2.3-22b-dev.safetensors` — `final` tier, 30 steps CFG 4.0. 46.1 GB bf16.
- `ltx-2.3-spatial-upscaler-x2-1.1` — optional finishing pass (Phase 5).

Pipelines actually used, all first-party Python via `python -m ltx_pipelines.<mod>` or direct import:
- `ti2vid_two_stages` / `distilled` → t2v AND i2v from one pipeline (`images=[]` vs `images=[first]`). This single-pipeline coverage is the main reason LTX beats Wan here.
- `keyframe_interpolation` → multiple keyframe images via additive guiding latents. This is the first+last / stitch-stills primitive.
- `retake` → regenerate an INTERIOR time region of an existing clip. Requires 8k+1 frames (97/193) and dims divisible by 32.

**Three corrections to the proposals, from the verified research — these change what we can promise:**
1. **There is NO video-extension/continuation pipeline in `ltx_pipelines`.** The docs were queried directly: "No pipeline is described for extending or continuing a video beyond its end." `retake` regenerates an interior region only. Therefore our `Continue` transition is NOT a native model capability — it is last-frame → next-shot-first-frame i2v chaining, which the long-form literature says visibly degrades after 3-4 hops (~15-20s). The UI must say so at the joint, and the API must enforce a max chain length established in Phase 0.
2. **`keyframe_interpolation` exposes NO per-image frame-index argument.** Its signature is `images: list[ImageConditioningInput]`; the widely-quoted README line about "keyframes at arbitrary frame indices" could not be found in the README or docs. First+last is safe to promise. Arbitrary-index placement is a Phase 0 verification item, not a design assumption.
3. **`ICLoraPipeline` can only be used with a distilled checkpoint**, and `LTX-2.3-22b-IC-LoRA-Ingredients` — the reference-sheet adapter that would have been the single best cross-shot identity primitive — is documented ONLY as a ComfyUI workflow (`LTXICLoRALoaderModelOnly`, `LTXAddVideoICLoRAGuide`); GitHub code search for "ingredients" in `Lightricks/LTX-2` returns 0 hits, and its official workflow is a single-stage dev+distilled-LoRA graph that `ic_lora.py` (two-stage, distilled-only) cannot express. **Out of scope by the no-ComfyUI hard rule.** Its absence is exactly why the FLUX entity-sheet approach below is load-bearing rather than optional. The repo is also gated (terms + marketing consent) — an ops annoyance worth knowing.

**Coherence engine — the existing FLUX.2 Klein backend (id 45, webbyg2). No new GPU.** Per-entity reference sets (4-6 stills: face close-up, body turnaround, two lightings; or product-style renders; or a clean location panel) beat whole-frame keyframe memory on identity accuracy by ~1.8x (EntityBench, Cohen's d +2.33). Stills are seconds to generate and trivially human-approvable. This is the highest-leverage coherence decision in the plan and it costs almost nothing.

**IMPLEMENTATION CAVEAT (locked decision #3):** two distinct FLUX jobs, decided in different phases. (1) *Keyframes/stills from text* — Klein does this today, and today's `DiffusionOutTranslator`/`CanonicalImageRequest` (text-to-image only: `model, prompt, n, size, quality, style, num_inference_steps, guidance_scale, seed`) is sufficient. This is all v1 needs. (2) *Auto "Generate variants" from one uploaded reference into a consistent sheet* (Phase 3) needs reference-conditioned generation, which requires BOTH new plumbing (a reference-image field threaded through the canonical image request + diffusion translator — it does not exist) AND a model that supports reference conditioning (**unverified for Klein**; FLUX.2 [dev] is the known-good fallback, swappable via `vid.keyframe_model`). Do not assume Klein delivers job (2); verify in Phase 3. v1 coherence therefore rests on user-uploaded references + LTX's own i2v/`keyframe_interpolation`, not on FLUX reference conditioning.

**Why LTX-2.3 over Wan2.2, licensing correction applied.** On a single H100 at identical 720p/121-frame settings the official LTX-2 paper (Table 1) measures 1.22 s/step for LTX-2 vs 22.30 s/step for Wan 2.2-14B — ~18x, from far more aggressive video-latent compression. With the 8+4 distilled schedule a 5s 720p clip is plausibly a sub-30s operation on an H200 versus many minutes for Wan. Wan2.2-A14B also wants ~80GB *with* `--offload_model True --convert_model_dtype` (the README's 80GB figure already assumes offloading), does 5s clips, and has no Wan2.2-generation FLF2V from the Wan team — so a Wan stack means mixing a 2.1-era model for anchored shots with a 2.2-era model for free shots, i.e. visible style mismatch between shots, which is precisely the property we need to preserve.

**Licensing: a note, not a driver.** LTX-2 Community License (2026-01-05) requires a paid commercial license above $10M aggregate annual revenue; U of I trips that threshold, but the operator's non-commercial internal-previz posture makes it note-and-monitor. The two clauses that *would* matter for a shared campus service were checked explicitly: (a) **no per-entity user cap and no geographic carve-out** — safe to expose to many campus users; (b) **machine-generated disclosure is required and cheap** — satisfied by a non-dismissible UI badge plus an ffmpeg metadata tag on every output. Attachment A also bars use in a product competing with Lightricks' offerings (they sell LTX Studio) — an internal gateway is not that, but it is worth a one-line record.

**HunyuanVideo-1.5 is EXCLUDED on deployability, not quality.** It is the cheapest good model (8.3B, 14GB with offload), but the Tencent Community License defines Territory as "the worldwide territory, excluding the European Union, United Kingdom and South Korea" and the exclusion binds **outputs**, not just weights. A campus service with international students, visiting scholars and EU collaborators cannot be gated on a per-user territory attestation. This is the one licensing pattern the correction says still matters, and it is disqualifying.

**Apache-2.0 fallback lane, config-swappable, no code change:** `Wan-AI/Wan2.2-TI2V-5B` (Apache 2.0, one model does t2v+i2v) drops into the same worker HTTP contract at a large quality/speed cost. `Wan-AI/Wan2.1-VACE-14B` via diffusers `WanVACEPipeline` (Apache 2.0, real Python API, `video`+`mask`+`reference_images` compose in one call, batch size hard-limited to 1) is the Phase 4 control-engine bake-off candidate. Neither ships in v1.

**Explicitly not in the stack:** Wan 2.5/2.6/2.7 (verified: HF full-text search for `Wan2.7` across ALL authors returns `[]` — the "Wan 2.7 open weights" sites are fabricated), HunyuanVideo-1.5 (territory), SkyReels-V3 (repo dead since 2026-01-30, successor V4 closed), Mochi/CogVideoX/Step-Video/Open-Sora (all dead; Open-Sora's 2026-04 "update" is a README-only commit, 0 code commits in 52 weeks).

## Serving design

**Node side — a new deploy artifact, `video-worker/`, modeled on the FLUX.2 Klein deployment (plain HTTP behind nginx, systemd unit, GPU-resident). It is ASYNCHRONOUS: submit/poll/fetch, not one long POST.** This is the single most consequential resolution between the two proposals (see decisionsForUser #1).

HTTP contract:
- `GET /health` → 200. **Must answer in <5s while a render is in flight.** `backend_health_timeout` is 5s and 3 consecutive failures flip the backend to UNHEALTHY mid-job. Generation therefore runs in a `ProcessPoolExecutor`/dedicated thread, never on the event loop. This is the most likely Phase 1 production bug and is an explicit Phase 0 acceptance criterion.
- `GET /v1/models` → `{"data":[{"id":"lightricks/ltx-2.3-distilled"}]}`.
- `GET /version` → engine version + checkpoint SHA.
- `GET /v1/capabilities` → the capability document `VideoAdapter` parses: `{supported_sizes, supported_durations, supported_fps, quality_tiers, pipelines:["t2v","i2v","keyframe","retake"], max_frames}`. This is what `GET /v1/videos/models` serves to the UI.
- `POST /v1/videos` → `202 {"id":..., "status":"queued"}`
- `GET /v1/videos/{id}` → `{status, progress, step, total_steps, eta_seconds, error}` — **real step-level progress falls out for free**, because the gateway polls rather than holding one socket. No callback endpoint with a shared secret is needed.
- `GET /v1/videos/{id}/content` → mp4 bytes, `Accept-Ranges: bytes`, 206 support.
- `DELETE /v1/videos/{id}` → cancel, free the GPU.

Serving config on the H200:
- Fully GPU-resident, **no offload**. 141GB holds the 22B DiT + Gemma-3-12B text encoder + VAE + upscaler simultaneously. Every offload mode in every stack is documented as memory-for-latency with no speed upside; SGLang's own cookbook says "Fastest single-GPU run when the model fits: disable CPU offload."
- `--quantization fp8-scaled-mm` with the fp8 checkpoints (real FP8 GEMM, Hopper-native). `fp8-cast` is memory-only and buys nothing here.
- FlashAttention-3. LTX's dispatch table in `ltx_core/model/transformer/attention.py` explicitly prefers FA3 on sm_90 ("Hopper (sm_90, H100): FA3 > FA4 > SDPA"), but FA3/FA4 are **optional try/except imports not installed by default**. **Assert at boot that FA3 actually loaded and log it** — the public RTX 5090 reference runs silently fell back to SDPA and left roughly 2x on the table.
- `torch.compile` warmed at process start over the **fixed preset matrix only**. This is the most important API-design consequence in the plan: torch.compile specializes to exact shapes, so free-form width/height/duration sliders would pay a multi-minute recompile per unique value inside a user's job. The UI exposes a menu; the gateway rounds or rejects off-menu values.
- Legal constraints from LTX itself: width/height divisible by 32, frame count divisible by 8 plus 1. Candidate presets (Phase 0 prunes any cell slower than ~3 min): `768x448`, `960x544`, `1280x704`, `704x1280`, `544x960`; frames `97`(~4s), `121`(~5s), `193`(~8s), `241`(~10s) at 24fps.
- `max_concurrent=1`. A single diffusion request drives a Hopper GPU to ~98.9% utilization; there is no KV cache and no batching upside. The usual 75%-of-capacity buffer rule does not apply — the buffer IS the gateway job queue. `check_hard_constraints`' `has_capacity` (`current_concurrent + queue_depth < max_concurrent`) is exactly the right gate at 1.
- Register via the admin API only (never direct DB): `POST /api/admin/backends/register {"name":"aspenN-gpuM-ltx","url":"http://aspenN:18300","engine":"video","max_concurrent":1,"node_id":N,"gpu_indices":[M],"gpu_memory_gb":141}`. Because `BackendRegisterRequest.engine` is typed as `BackendEngine`, adding the enum member makes the existing endpoint accept video backends with zero endpoint changes.
- GPU telemetry comes from the existing per-node sidecar on `:8007` sliced by `Backend.gpu_indices`, exactly as FLUX does. The worker needs no `/metrics`. Register with the correct `node_id`/`gpu_indices` or the scorer's utilization term degrades to flat half-credit.
- nginx vhost in front: `client_max_body_size 128m`, `proxy_read_timeout 900s`.

**Gateway side — a real `VideoAdapter`, not VLLMAdapter.** I verified `BackendRegistry._create_adapter` (registry.py:869) falls through to `VLLMAdapter` for anything non-Ollama. That would technically work, but `VLLMAdapter._get_models` derives capabilities from name substrings ("vl", "vision", "70b", `max_model_len`) which produce garbage for a video checkpoint — a model named `...-vl-...` would be tagged multimodal. Since we are already adding a `BackendEngine.VIDEO` enum member (migration required regardless), adding an explicit adapter branch is trivial and buys us the real capability document that `/v1/videos/models` and the UI both need.

**Why not SGLang Diffusion.** It would give an OpenAI-shaped `/v1/videos` for free and map 1:1 onto the registry, but its compatibility matrix lists LTX-2.3 with "no dedicated optimization" and batching "?" (untested), its documented HTTP surface is only `prompt`/`size`/`seconds`, and none of the pipelines we actually depend on (keyframe conditioning, retake) are reachable over it. A ~400-line FastAPI wrapper we own is the boring choice. Listed in decisionsForUser as the biggest reversible decision.

**`_proxy_with_retry` is NOT modified.** Video never enters it. Separately, I confirmed a latent bug worth fixing on its own merits: `_proxy_image_request` (inference.py:1777) computes `timeout = max(backend_request_timeout_per_attempt, 300.0)`, but `_proxy_with_retry` wraps the call in `asyncio.wait_for(..., timeout=180)` at line 1329-1331 — **the inner 300s is dead code and every image render is silently cancelled at 3 minutes, then retried on up to 2 more GPUs.** That is a real defect in the shipped image path; it should be filed independently of this feature.

## Job & orchestration model

**Fully async, gateway-orchestrated, three levels: project → shot → render-attempt.** Nothing in the video path touches `InferenceService._proxy_with_retry`.

States (`video_jobs.status`): `queued → planning → rendering → assembling → completed | failed | cancelled`. Per-shot (`video_shots.status`): `pending → queued → rendering → rendered | failed | skipped`. A shot already `rendered` and unchanged is never re-rendered — this is what makes "regenerate shot 7" cheap and is the entire justification for the hard-cut architecture.

**Runner** (`backend/app/services/video_runner.py`), started from `main.py` lifespan behind `settings.video_runner_enabled`, alongside `_retention_loop`. Each tick:
- Claim `queued` jobs. Use MariaDB row semantics for multi-worker safety: `UPDATE video_jobs SET status='planning', claimed_by=:worker_id, started_at=NOW() WHERE status='queued' AND id=(SELECT id FROM (SELECT id FROM video_jobs WHERE status='queued' ORDER BY priority DESC, id ASC LIMIT 1) t)` then check `rowcount==1`. No Redis lock required; the flag exists so the loop can be pinned to one process if anything looks wrong.
- Walk shots in dependency order. A shot with `transition='continue'` cannot start until its predecessor is `rendered` (it needs the predecessor's last frame). Shots joined by `cut` are independent and fan out in parallel across all healthy video backends.
- **Route each shot through the existing machinery, honestly**: `create_job_from_video_shot` builds a `Job(modality=JobModality.VIDEO_GENERATION, is_streaming=False)`; `registry.get_backends_with_model(model, Modality.VIDEO_GENERATION)` filters candidates; `BackendScorer` ranks them. `check_hard_constraints` needs no change — `supports_modality` falls into the `else: True` branch (verified), and modality isolation is enforced upstream by `crud.get_backends_with_model`'s `Model.modality ==` filter.
- Do **not** call `scheduler.wait_for_capacity` with its current `backend_request_timeout/2` = 150s deadline. Runner semantics are "leave the shot `queued`, retry next tick," bounded by `video_job_max_wall_seconds`.
- Submit to the worker, poll every `video_runner_poll_interval_seconds` (default 5), chain last-frame → next first-frame for `continue`, trigger assembly, fire webhooks.

**Retry policy is inverted from LLM.** `backend_retry_max_attempts=3` on a 10-minute render is up to 30 wasted GPU-minutes. Per-shot retries cap at 2, only on connect-error or 5xx-at-submit — never on a render that started and then failed (that gets surfaced) — with backoff, and never re-routed inside the same tick. `BackendRouter._gc_stale_jobs` / `_cleanup_db_orphans` thresholds must be raised or VIDEO_GENERATION exempted, or legitimately long renders get reaped mid-flight.

**Progress is real, not theater.** Two sources merged: the worker's per-shot `{step, total_steps}`, and the gateway's `shots_rendered / shots_total`. Job progress = `(completed_shots + current_shot_fraction) / total_shots * 0.9 + assembly_fraction * 0.1`. The Images tab's `Math.min(95, elapsed/130*100)` fake bar is explicitly not carried over. ETA seeds come from a per-`(model, size, frames, quality)` rolling median of the last 20 completed shots, with Phase 0 measurements as the cold-start defaults.

**Delivery.** Poll (`GET /v1/videos/{id}`, documented 5-10s with backoff), SSE (`GET /v1/videos/{id}/events`, what the dashboard consumes so it does not hammer the poll endpoint), and optional `callback_url` webhook with HMAC-SHA256 and idempotency key `{job_id}-{status}`. No partial-video streaming — nobody in the industry does it and it is not worth inventing.

**Assembly runs on a separate bounded pool from the GPU:** global `asyncio.Semaphore(settings.video_assembly_workers)` (default 2, ~cores/4 for x264), ffmpeg as a subprocess with `-nostdin -y -progress pipe:1`, per-job scratch dir, process kill on cancel. Never inside a request handler — same failure class as the OCR worker OOM and the v2.4.0 orphaned-generation bug.

**Cancellation propagates:** `DELETE /v1/videos/{id}` sets `cancel_requested`; the runner sends `DELETE /v1/videos/{worker_job_id}` to the worker, kills in-flight ffmpeg, marks remaining shots `skipped`.

**Crash recovery — the payoff for the async-worker spine.** On startup the runner re-adopts `rendering` jobs whose `heartbeat_at` is older than `video_job_stale_heartbeat_seconds` (120): it re-polls the worker by `video_shots.backend_job_id` and either resumes tracking or re-queues. A gateway restart mid-render costs nothing; the GPU keeps working. Under the alternative (gateway holds the render), a restart orphans an H200 for the length of the render.

**Quota — record the truth, charge through the existing meter.** Video must not be token-metered honestly (image generation is effectively free today: `estimated_completion_tokens=0`, diffusion returns no `usage`), but a new quota dimension means touching every existing report. So: always record `video_jobs.gpu_seconds` and `duration_seconds` as first-class columns (the honest unit, preserved for history), AND charge a derived token-equivalent against `Quota.tokens_used` so enforcement works today with zero new quota machinery: `cost = vid.token_cost_per_second * duration_seconds * quality_multiplier * resolution_multiplier`. Pre-check at enqueue against remaining quota (so a user cannot queue 40 jobs they cannot afford), deduct at completion. Also enforce `vid.max_concurrent_jobs_per_user` (default 1) — the real fairness lever on a one-render-at-a-time resource. If a real GPU-seconds budget is added later, the history is already there.

## API surface

- POST /v1/videos — API-key auth, returns 202 with the job object (never the video). Two accepted bodies on one route. (a) Simple: {model, prompt, seconds:"5", size:"1280x704", fps?:24, quality?:"draft|standard|final", seed?, negative_prompt?, input_reference?:{image_url|file_id|b64_json}, frame_images?:[{image_url|file_id, frame_type:"first_frame"|"last_frame"}], input_references?:[{image_url|file_id, reference_type:"asset"|"style"}] (max 4), generate_audio?:bool, callback_url?}. (b) Storyboard: {model, storyboard:{style_prompt?, reference_images?:[...], fps, size, quality, seed_policy, shots:[{prompt, seconds, transition:"cut"|"crossfade"|"continue", first_frame?, last_frame?, input_references?, seed?, camera?}]}}. Simple bodies are stored as a one-shot project so there is exactly one execution path.
- Job object (echoed by every endpoint): {id:"vid-<24hex>", object:"video", status:"queued|in_progress|completed|failed|cancelled", progress:0-100, created_at, started_at, completed_at, expires_at, eta_seconds, model, size, seconds, fps, quality, prompt, error:{code,message}|null, content_url, usage:{duration_seconds, gpu_seconds, token_equivalent}, shots:[{index, status, progress, seconds, content_url, poster_url}]}. Deliberately OpenAI-shaped — uses `in_progress` (not LiteLLM's `processing`) so stock SDKs work, accepting `processing` on input for tolerance. `shots[]` is the one bespoke addition.
- GET /v1/videos/{id} — poll. Documented 5-10s interval with backoff. Returns 404 (not 403) for another user's job id — no existence leak.
- GET /v1/videos/{id}/content?variant=video|poster|shot&index=N — StreamingResponse with `Accept-Ranges: bytes` and 206 Partial Content so browser <video> scrubbing works. NEVER a single in-memory Response body and never base64 for the full render — that is the images path's mistake (`images_serve` loads the whole file via `storage.retrieve()` into one Response) and it does not scale past a few MB.
- GET /v1/videos — list caller's jobs. Query: status, limit (<=50), offset. Returns {data:[job...], total}.
- DELETE /v1/videos/{id} — cancel if running (propagates to the worker and kills ffmpeg), delete artifacts + rows if terminal.
- POST /v1/videos/{id}/shots/{index}/regenerate — {prompt?, seed?, first_frame?, ...}. Invalidates ONLY that shot plus any downstream shot joined by transition="continue", then re-assembles. Highest-value endpoint in the design; falls out free from the hard-cut architecture and is what makes long-form iteration economically viable.
- GET /v1/videos/models — capability discovery. Per-model {id, supported_sizes, supported_durations, supported_fps, supported_qualities, max_shots, max_total_seconds, supports_keyframes, supports_reference_images, supports_audio, max_continue_chain, license_notice:"AI-generated"}. Served from the worker's /v1/capabilities via VideoAdapter. The UI renders its controls FROM this, so the preset matrix lives in exactly one place and always matches the torch.compile warm set. Both independent proposals converged on needing this — strong signal.
- POST /v1/videos/assets — multipart upload of a reference image/still. Returns {file_id, sha256}. Keeps base64 out of request bodies and gives the vision policy judge something to scan before it is ever used as conditioning.
- GET /v1/videos/{id}/events — text/event-stream of {status, progress, shot_index, message}. Consumed by the dashboard so SSE stays out of the poll path. nginx needs proxy_buffering off on this location.
- Dashboard (session cookie, separate router, mirrors dashboard/images.py): GET /video (page), POST /video/api/check-policy, POST /video/api/projects, PATCH /video/api/projects/{id} (edit shot list), POST /video/api/projects/{id}/render, GET /video/api/jobs/{id}, GET /video/api/jobs/{id}/events, POST /video/api/assets, GET /video/api/library?offset=&limit=12, GET /video/serve/{video_id} (Range-capable), DELETE /video/delete/{video_id}.
- Admin: GET/POST /admin/video-config (all vid.* keys + per-user video_generation_enabled toggle, sidebar active=='video_config'). GET /api/admin/video-jobs (queue depth, running, failed-24h, per-backend) for ops visibility.
- DEFERRED, not in v1: POST /v1/videos/{id}/extend. It would be sugar over 'append a shot with transition=continue' — and since LTX has no native continuation pipeline, it carries the same 3-4 hop drift ceiling. Ship it in Phase 5 only if the shot API proves clumsy for that use.

## UI design

**One page, `/video`, three panes, neutral vocabulary throughout.** The nouns are Project, Shot, Reference, Timeline, Render — never campaign, brand, client, script, or spot. A researcher animating a protein and someone mocking up a 45-second promo see the same tool. Bootstrap 5 + vanilla JS in `{% block extra_js %}`, same as every other page — no framework, no build step. Page CSS goes inline at the top of `{% block content %}` and MUST consume `var(--mr-*)` tokens so dark mode works (images.html hardcodes its indigo/purple accents and does not adapt — do not repeat that).

**Left rail — References (the entity bank).** A grid of uploaded or FLUX-generated stills, each with an optional free-text label ("Dr. Chen", "the enzyme", "north campus at dusk", "the mug"). Labels are just strings; nothing in the system knows what a "product" is. Two actions: *Upload* and *Generate variants* — the latter calls FLUX.2 through the existing image pipeline to produce a 4-6 image turnaround set from one upload. This is the highest-leverage coherence feature in the plan and costs no new GPU. References can be marked project-wide (applied to every shot) or per-shot. Every upload is policy-scanned by the vision judge, cached by sha256, before it becomes usable.

**Center — Shot editor.** A vertically stacked, drag-reorderable list of shot cards. Each card: prompt textarea, duration stepper (from the allowed-durations enum), a first-frame slot and an optional last-frame slot (drop a reference, generate a keyframe with FLUX, or "use previous shot's last frame"), a reference chip row, an optional camera hint dropdown (static / slow push in / pull out / pan left / pan right / orbit — plain language, mapped to prompt suffixes), and a seed field. The **transition selector lives on the joint between cards**: **Cut** (default), **Crossfade**, **Continue**. Continue is marked with an explicit warning that it is last-frame chaining, is more expensive, and drifts after ~3 hops — with the chain length capped by `vid.max_continue_chain` and the link shown as an editable choice, not magic. Each card shows its own status chip and, once rendered, an inline `<video>` preview plus **Regenerate this shot**. Regenerating shot 7 does not touch 1-6; if shot 8 is joined by Continue, the UI says so and offers to re-render the chain.

**Right — Project settings + Render.** Size, fps, quality tier, global style prompt, seed policy (fixed / per-shot), audio toggle, and a live total-duration readout against `vid.max_total_seconds` ("38s / 90s"). **All controls render FROM `GET /v1/videos/models`**, so the UI can never offer an uncompiled shape and the preset matrix has exactly one source of truth. No free-form width/height sliders — that is the torch.compile constraint made visible, and it is the better UX anyway. Quality tiers are described in plain terms ("Draft — fast, for checking framing") with an inline estimate derived from the rolling-median ETA. An estimated-cost line ("≈ 16,000 tokens of your quota") appears before the user commits, because video is expensive and the cost should not be a surprise.

**Progress is real, not theatrical.** The page opens an EventSource on `/video/api/jobs/{id}/events`. The timeline strip fills shot-by-shot; the current shot shows the worker's actual `step/total_steps`; queue position and honest remaining time are displayed. The Images tab's `Math.min(95, elapsed/130*100)` fake bar and its 8s/25s/45s reassurance-copy rotation are explicitly not carried over. The page is safe to leave — the job is a durable row and reopening `/video?job=<id>` reattaches.

**Output.** Inline `<video controls preload="metadata" poster="...">` served from `/video/serve/{id}` with Range support so scrubbing works. Download, per-shot download list, "Use as first frame" (feeds a last frame back into the shot editor — the manual coherence loop), and Regenerate with same seed. A **persistent, non-dismissible "AI-generated" badge** on the player: it satisfies the LTX license's machine-generated-disclosure obligation and is also just correct.

**Library.** Paginated grid (`GET /video/api/library?offset=&limit=12`) of **poster JPEGs, not autoplaying video elements** — do not lazy-load mp4s in a grid. Hover shows duration/size/model; click opens a modal player. Real `<button>` elements with aria-labels, never `<span onclick>`. A visible storage-usage bar against the per-user cap.

**Ad previz, deliberately demoted to one dropdown.** "Start from a template" offers neutral shot skeletons — *Object/subject showcase (6 shots)*, *Explainer (intro / three points / recap)*, *Event promo*, *Process or timelapse*, *Interview B-roll*, *Single take* — each of which merely pre-populates the shot list with empty prompts and sensible durations. A thin layer over the general primitives, never the spine.

**Accessibility, enforced before anyone writes a video-specific test.** `test_accessibility.py` uses `ALL_TEMPLATES = TEMPLATE_DIR.rglob("*.html")` as a params fixture (verified at line 79-85 and again at line 434), so both new templates are auto-enrolled and must pass on day one: unique ids, `<th scope>` and `<caption>` on tables, `aria-hidden` on decorative `bi-*` icons, `aria-label` on every icon-only button, admin templates including `_sidebar.html`, an `<h1>` on standalone pages. Beyond the generic sweep: `role="status" aria-live="polite"` on the render-status region (long waits MUST be announced — the images overlay is silent, a real defect), `aria-modal`/`aria-labelledby` on modals, real `role="radiogroup"` button groups with keyboard support for the size/quality/transition chips (images.html uses click-handled `<span class="size-btn">` — do not clone), and `<video controls>` with an accessible name. Gallery cards are built by JS and are therefore invisible to the regex-based linter — build them correctly anyway.

## Risks

- **NO PRIMARY H100/H200 BENCHMARK FOR LTX-2.3 EXISTS.** Every published latency number is RTX 5090 or extrapolated, and Lightricks publishes NO VRAM table, no resolution/duration matrix and no timings in either the repo or the model card (official docs say only 'minimum 32GB+, recommended A100 80GB or H100'). The capacity plan, ETA defaults and preset matrix all rest on Phase 0. If a 5s 720p clip costs 3 minutes rather than 30 seconds, the preset matrix shrinks and per-user caps drop — but nothing architectural changes, which is exactly why Phase 0 comes first.
- **ONE H200 SERVES ROUGHLY ONE GENERATION AT A TIME.** A single diffusion request drives a Hopper GPU to ~98.9% utilization; there is no KV cache and batching is off by default in every serving stack for that reason. Realistic steady state is a handful of genuinely concurrent interactive users before queue wait becomes unacceptable, and a 16-shot project monopolizes the GPU for many minutes. Without vid.max_concurrent_jobs_per_user=1, one user blocks the campus. Mitigations: max_concurrent=1, per-user cap, visible queue position and honest ETA, and documentation that sets expectations. Do not let anyone expect chat-like responsiveness.
- **NAIVE LAST-FRAME CHAINING VISIBLY DEGRADES AFTER 3-4 HOPS** (~15-20s at 5s/clip): identity drift, then frozen frames or periodic motion, contrast buildup, motion collapse. Compounded by the verified finding that **LTX has no native video-continuation pipeline at all** — 'Continue' is i2v chaining, full stop. The hard-cut-by-default architecture is the mitigation, but users will reach for Continue and be disappointed. The UI must warn at the joint, the API must enforce vid.max_continue_chain, and Phase 4 must establish that number by measurement rather than folklore.
- **HEALTH-CHECK FLAPPING DURING RENDER.** backend_health_timeout is 5s and 3 consecutive failures mark the backend UNHEALTHY. A worker that blocks its event loop during generation gets evicted from routing mid-job. Generation MUST run off-loop in a ProcessPoolExecutor. This is the most likely Phase 1 production bug and is an explicit Phase 0 acceptance criterion.
- **torch.compile SHAPE BRITTLENESS.** Any resolution/frame-count outside the boot-warmed bucket set triggers a multi-minute recompile inside a live user's job. Mitigation: the preset matrix is the ONLY legal input, it lives in one file on the node, is mirrored into vid.allowed_sizes, and the UI renders its controls from GET /v1/videos/models. The tradeoff — 'user control over size and length' is satisfied by a curated warm menu, not a slider — must be stated plainly in the docs.
- **VIDEO WILL FILL /archivedb.** The prod artifact mount already had a disk-pressure incident; user_images has NO retention today (verified: retention.py _DEFAULTS covers request_images/responses_store/conversations and nothing else); video files are 50-500x larger, plus per-shot intermediates, plus normalized near-lossless assembly scratch, plus reference sets. retention.user_videos_days and a per-user storage cap must ship in PHASE 1, enforced at job-submit (fail fast, not mid-render), not deferred the way images retention was.
- **THE ENUM ALTER ON `requests` IS THE RISKIEST DDL IN THE PLAN.** `requests` is the largest table in prod. Appending a 9th ENUM value keeps storage at 1 byte so ALGORITHM=INPLACE, LOCK=NONE should apply — but state it EXPLICITLY in the DDL rather than trusting the server, and dry-run against a restored copy. MariaDB DDL is non-transactional, so a failure mid-migration leaves partial state requiring manual cleanup.
- **QUOTA IS A FUDGE UNTIL IT ISN'T.** Charging GPU-seconds through Quota.tokens_used avoids a schema change but makes video usage indistinguishable from LLM usage in every existing report and dashboard. Mitigated by recording video_jobs.gpu_seconds and duration_seconds separately so a real GPU-seconds dimension can be added later without re-deriving history — but until then, the most expensive modality on the cluster is invisible to the accounting that governs every other modality.
- **REFERENCE IMAGES BYPASS MODERATION TODAY.** image_policy.evaluate_prompt is TEXT-ONLY and fail-closed; the image path never inspects conditioning stills. For a shared campus service where users upload photographs of real people, a vision judge is not optional — and likeness law (ELVIS Act, CA AB 1836/2602, the pending NO FAKES Act, which advanced out of Senate Judiciary in June 2026) targets exactly this and reaches tool providers. Phase 1 requires a recorded consent attestation for any upload containing a person; Phase 3 adds the vision judge. Note also that image_policy._call_judge uses verify=False with a 30s read timeout and bypasses the scheduler — acceptable amortized over a multi-minute render, unlike chat.
- **XFADE OFFSET MATH IS A CLASSIC SILENT FAILURE.** ffmpeg's xfade requires constant frame rate and identical resolution, pixfmt, fps and timebase on both inputs, and chaining N clips requires cumulative offset arithmetic where an off-by-one truncates or freezes a clip with NO error. No maintained Python library does it correctly. Mitigation: the canonical-intermediate normalization step makes the constraints true by construction, and the offset formula is exhaustively unit-tested at N=2/3/16 before any ffmpeg ever runs.
- **ffmpeg BUILD FLAGS ARE A SILENT FAILURE MODE.** Verified absent from the Dockerfile entirely. Beyond installing it, libx264 and libass are load-bearing and a stock distro static build may lack them. Assert `ffmpeg -buildconf` at startup and fail loudly rather than discovering it inside a user's render.
- **COPY-PASTE OF THE IMAGES SPLIT-BRAIN.** Verified: v1_openai.py and dashboard/images.py duplicate the guardrail block and already disagree (steps 20 vs 30, max_n 4 vs 1, dims 1024 vs 2048). If video repeats this it doubles immediately. Both entry points must go through video_in.py, and a test asserts identical canonical output from identical bodies.
- **MULTI-WORKER CLAIM RACE.** Multiple uvicorn workers each running the runner loop could double-claim. Mitigated by the atomic UPDATE...WHERE status='queued' + rowcount check (no Redis needed), but it MUST be exercised under a simulated race in test_video_jobs.py, and video_runner_enabled exists so the loop can be pinned to one process if anything looks wrong.
- **LTX-2.3 IS COMMUNITY-LICENSED WITH A FAST SUCCESSOR CADENCE** (2.0 → 2.3 in two months, 2.0 deprecated on the hosted API, no tagged GitHub releases). Mitigation: the model is a config value behind a translator boundary, the checkpoint SHA and ltx_pipelines commit are pinned in requirements.txt, and Wan2.2-TI2V-5B (Apache 2.0) drops into the same worker HTTP contract if the license posture ever changes.
- **TRAINING-DATA PROVENANCE IS UNDISCLOSED FOR EVERY OPEN-WEIGHTS OPTION.** For internal non-commercial previz this is an accepted risk — but it should be accepted ON THE RECORD, and it is the reason not to let outputs leave the institution without the AI-generated label and a per-generation log (model, license, prompt, reference hashes, attestation).

## Decisions for you

- **1. THE BIG ONE — where the render waits. RESOLVED 2026-07-22: async worker (option b), ratified.** Original analysis retained below. Option (a): the gateway holds the render, reusing `_proxy_with_retry` with a new `timeout_override` parameter to escape the 180s `asyncio.wait_for`. Minimum new machinery, reuses a proven path. Option (b) — MY RECOMMENDATION: the WORKER is async (submit/poll/fetch), the gateway runner polls, and `_proxy_with_retry` is never touched. Four reasons (b) wins: it changes zero lines in the function that handles 100% of LLM traffic; a gateway restart mid-render costs nothing because the runner re-adopts by `backend_shot_job_id` instead of orphaning an H200; real step-level progress falls out for free instead of needing a node→gateway callback with a shared secret; and video-appropriate retry semantics (2 attempts, never re-route mid-tick) don't have to be contorted into a function whose LLM semantics are 3 backends × 180s. The cost of (b) is ~150 more lines in the worker and one more state column. **Separately and regardless of this choice**: `_proxy_image_request`'s 300s timeout is dead code today — the outer 180s wait_for kills every image render at 3 minutes and retries it on up to 2 more GPUs. That is a shipped bug worth its own ticket.
- **2. HOW AMBITIOUS v1 IS. (The other genuine conflict.)** One proposal ships single-clip with a UI in Phase 1 and defers storyboard; the other ships plumbing → storyboard API → UI, meaning no user sees anything until Phase 3. MY RECOMMENDATION: the graft above — UI in Phase 1 (single-clip t2v/i2v/first+last is independently useful to most of the stated audiences), but write the FULL four-table schema in the Phase 1 migration so Phase 2 is additive code rather than a schema redo. If you need a 30-60s deliverable on day one, Phases 1 and 2 merge and the timeline roughly doubles — say so now, because it changes the Phase 0 exit criteria.
- **3. SERVING RUNTIME.** (a) First-party FastAPI worker over ltx_pipelines (~400 lines we own): full access to keyframe/retake pipelines, exact control of the /health contract and the async job semantics this design depends on. ## Appendix A — Completeness critique

The critique below was produced against the plan above and is **not yet folded into it**. Its P0 items are genuine corrections, especially #1 (user-supplied clips).

Verified against the tree: migration head is `064`; nginx is `client_max_body_size 50m` / `proxy_read_timeout 720s`; Dockerfile has no ffmpeg; `validate_request_fields` has exactly one non-test call site (`openai_in.py:53`); `retention.py::_DEFAULTS` has no user-media key; `inference.py:1331` wraps attempts in `asyncio.wait_for(..., 180s)`. Those plan claims hold. What follows is what does not.

---

## P0 — breaks a stated requirement or will fail in production

**1. Capability #3 is not delivered: user-supplied CLIPS cannot enter the timeline.**
The requirement is "stitching multiple images *and clips*". The plan supports stills as conditioning and concatenation of *model-generated* shots, but there is no path for a user to place an existing mp4 on the timeline. `POST /v1/videos/assets` is described as "multipart upload of a reference image/still"; `video_assets.kind` is `('reference','keyframe','upload','shot_output','final')` with no source-clip member; the storyboard shot object has no `source_asset_id`; `video_policy.py` scans images only. **Fix:** add `kind='source_clip'`; accept video uploads (ffprobe-validated: codec/duration/resolution allow-list, `vid.max_upload_clip_seconds`); add shot `type: "generated" | "source"` where a `source` shot carries `{asset_id, trim_start, trim_end}`, is skipped by the runner's GPU path entirely, and is handled by `normalize_clip()` like any other segment; extend the vision judge to sample N frames of an uploaded clip. This also makes "trim and re-cut an existing recording" work, which is the single most-requested general-purpose video operation on a campus.

**2. `_cleanup_db_orphans` will mark every video request FAILED after 5 minutes.** Verified at `routing.py:614`: it selects `Request.status == QUEUED AND created_at < now-5min`, filters against `queue.get_all_jobs()`, and fails the rest. The plan addresses `_gc_stale_jobs` but never mentions this one. A video `requests` row (which `video_jobs.request_id` FKs to) sits in a non-completed state for the whole multi-minute render while the per-shot scheduler Job's `request_id` is *the shot's*, not the parent request's — so the parent is invisible to the scheduler queue and gets reaped on the first sweep. **Fix:** add `.where(Request.modality != Modality.VIDEO_GENERATION)` to the orphan query, or give video requests a distinct status transition at enqueue and exclude it explicitly. Add a `test_video_schema_contract.py` assertion that the exclusion is present.

**3. Nobody calls `on_job_started` / `on_job_completed` for video shots.** The plan routes each shot through `scheduler.submit_job`/`route_job` and correctly notes `has_capacity = current_concurrent + queue_depth < max_concurrent` is the gate at 1 — but never states who releases the slot, and the async spine means the natural releaser (`_proxy_with_retry`'s finally block) is gone. Consequences: `max_concurrent=1` stops gating after the first shot; `_backend_queue_depths` leaks monotonically; crash re-adoption restores the *shot* but not the in-memory depth counter, so a restarted gateway permanently believes the GPU is busy until the 15-minute GC sweep. **Fix:** the runner owns the full lifecycle explicitly — `on_job_started` at worker-submit-accept, `on_job_completed`/`on_job_failed` in a `finally` around the poll loop, and a re-adoption path that re-increments depth for shots it resumes. Unit-test the leak: submit → simulate runner crash → re-adopt → assert depth == 1, not 2.

**4. `video_worker_timeout_seconds: int = 60` is wrong for the artifact fetch.** The setting is documented as "per-HTTP-call to the worker (submit/poll/cancel are sub-second)" — but `GET /v1/videos/{id}/content` is also a call to the worker and moves 50–300MB. At 60s it will abort on any congested link. The plan also never says *who* moves the mp4 from node to gateway, when, or where the gateway's copy lands relative to `video_store.store_stream`. **Fix:** split `video_worker_fetch_timeout_seconds` (default 900) from the control-plane timeout; specify the transfer explicitly as "runner streams worker→`video_store.store_stream` on shot completion, deletes worker-side artifact via `DELETE /v1/videos/{id}` after checksum match"; add `sha256` to the worker's status payload and verify it. Both settings need docker-compose lines.

**5. Reference images ship unguarded in Phase 1 while the vision judge lands in Phase 3.** Phase 1 explicitly ships i2v plus a References rail with uploads; the moderation for those uploads is Phase 3. That is the highest-liability surface (identifiable likeness, animated) shipping with a checkbox. One VLM call per upload, cached by sha256, is a few hundred lines and the cluster already serves multimodal models. **Fix:** move `evaluate_reference_image()` into Phase 1; it gates the *upload* endpoint, not the render, so it costs nothing at render time. Additionally: no output-side check exists anywhere — the plan judges the prompt and (eventually) the input, but never a frame of what was produced. Add a sampled-frame check on the final render behind `vid.output_review_enabled` (default on, 3 frames, cached), at minimum for jobs that used a person reference.

**6. Accessibility: this is a public university shipping an audiovisual media tool with no captions story.** The plan's a11y section is thorough about the *UI chrome* and silent about WCAG 1.2.1/1.2.2 for the *content*. `generate_audio` is in the v1 API surface and in `video_projects`. A generated clip with speech and no caption track is a Section 508 defect for the institution, and the plan's own decision #8 says audio should be off through Phase 4 — so v1's API accepts a parameter the design says not to use. **Fix:** (a) reject `generate_audio` with a 400 until the phase that supports it, rather than accepting-and-ignoring; (b) when audio ships, require either a `<track>` sidecar (`.vtt`) or an explicit "no speech content" attestation before download, and add a "Generate captions" action wired to the existing STT backend; (c) label silent renders as such in the player's accessible name.

## P1 — design defects and internal contradictions

**7. The `inference.py` change contradicts the architecture.** `filesChange` says the only edit is extending `_create_request_record`'s parameter allow-list "so video params appear in the audit trail" — but `_create_request_record` is a private method of `InferenceService`, which the plan says video never touches, and `testPlan` asserts `_proxy_video_request` does *not* exist. So either video calls into InferenceService (contradicting the thesis) or the edit is dead. **Fix:** decide explicitly — create the audit row in `video_jobs.py` via `crud.create_request(...)` directly, drop the `inference.py` edit entirely, and add a contract test that `video_api.py` does not import `InferenceService`. Separately, note that bypassing InferenceService also bypasses `_complete_request`'s DLP enqueue and `update_api_key_usage` — the plan mentions neither. Enumerate every side effect of `_complete_request` and state which ones video re-implements and which are deliberately dropped.

**8. Field validation is top-level-only; the storyboard body is nested.** `validate_request_fields` iterates the raw dict's keys. A typo inside `storyboard.shots[3]` (`"duration"` instead of `"seconds"`, `"steps"`, `"image"`) is silently dropped — precisely the class of bug v2.8.2 was built to eliminate, reintroduced at greater depth. **Fix:** define `VIDEO_SHOT_ACCEPTED`/`_IGNORED`/`_HINTS` and have `build_canonical_video_request` call `validate_request_fields` once per shot dict with `dialect='video.shot'` and a `param` path of `storyboard.shots[3].duration`. Test it.

**9. Quota pre-check does not reserve, so it does not bound anything.** "Pre-check at enqueue against remaining quota … deduct at completion" — N submissions in the same window each see the same remaining balance and all pass. With `max_concurrent_jobs_per_user=1` this is limited but not closed (a user can queue 40 jobs sequentially over a minute). Also `vid.token_cost_per_second = 2000` is an unbenchmarked constant that makes a 60s final-tier render ~500k+ tokens after multipliers — plausibly larger than a whole role's default quota, meaning the feature is unusable on day one for standard users and nobody will notice until a pilot user files a ticket. **Fix:** reserve at enqueue (debit `Quota.tokens_used`, refund on `failed`/`cancelled`, reconcile on completion); and make Phase 0 output the actual GPU-seconds per preset so `token_cost_per_second` is set from measurement, with a documented worked example ("a 30s standard 720p render costs ≈ X, a student's monthly quota is Y") checked against the real default quotas before Phase 1 ships.

**10. Per-user boolean gating does not scale to a campus service.** Images uses a per-user toggle and the plan clones it. For "a shared campus service with many users" — the framing the licensing analysis itself leans on — an admin flipping checkboxes one at a time is the wrong control plane, and the admin page's paginated user table makes it worse. **Fix:** add group-level enablement (`Group.video_generation_enabled`, or `vid.enabled_group_ids` in app_config) with the per-user boolean as an override, and expose both on `/admin/video-config`. Effective access = group default OR user override. This is cheap now and a migration later.

**11. `vid.enabled` is seeded `True` in migration 067.** Deploying the migrations turns the feature on. Combined with rule #6 in memory (always confirm before prod deploy), the safe posture is off-by-default with an explicit admin action. **Fix:** seed `vid.enabled = False`; document the enablement step in the runbook. Related: the plan has **no deployment section at all** — no prod migration runbook, no `alembic upgrade head` step inside the container, no `/archivedb/video` host-dir creation, no rollback plan for a half-applied non-transactional MariaDB DDL, no statement about `VIDEO_WEBHOOK_SIGNING_KEY` landing in `/opt/mindrouter/.env`. Add one.

**12. "Hyperparameter control" is materially reduced, and the mitigation is a shrug.** The requirement names size, length, quality "and other hyperparameters"; the plan deletes steps, guidance, scheduler and sampler entirely and constrains size/length to a compiled menu. The risks section states this plainly, which is good, but the design offers no path back. **Fix:** two concrete escapes, both cheap: (a) mark the frame dimension dynamic at compile time (`torch._dynamo.mark_dynamic`) so *length* is free within a compiled resolution — Phase 0 can measure whether this costs anything; (b) an "Advanced / uncompiled" quality tier that runs eager with free-form `width`/`height`/`num_inference_steps`/`guidance_scale`, rate-limited and labeled "slower, no ETA". Expose `seed` and `negative_prompt` unconditionally (they don't trigger recompiles). Without at least one of these, the plan should say outright that the requirement is partially descoped.

**13. Failed-job and worker-side garbage is never reconciled.** The plan covers gateway-side re-adoption but not the inverse: worker restarts and forgets a job (what does the runner do with a 404 on poll?); a job row is deleted while the worker still renders (orphaned H200); ffmpeg killed mid-assembly leaves a scratch dir; a failed 12-shot job leaves 11 rendered shot mp4s in storage for the full 90-day retention. **Fix:** (a) worker exposes `GET /v1/videos` listing its jobs; the runner reconciles on startup and `DELETE`s anything the gateway doesn't know about; (b) define poll-404 as terminal-failure-with-requeue-eligible; (c) scratch dirs created under a per-job path reaped by a `finally` and by a startup sweep; (d) separate `retention.video_shot_intermediates_days` (default 7) from the final-render retention, and delete normalized intermediates immediately after successful assembly.

**14. Storage sizing is asserted, not computed, on a disk that has already filled.** `20GB/user` × an unspecified user count, with no global watermark. Per-user caps do not bound the pool. **Fix:** state the available `/archivedb` capacity, compute a worst case (users × cap), and add global admission control: `vid.pool_soft_limit_gb` / `vid.pool_hard_limit_gb`, checked at enqueue, with the hard limit refusing new jobs and surfacing on the admin queue page. Also state what the "final" tier + x2 upscaler does to per-render size — that is the multiplier that will actually blow the budget.

**15. GPU contention is hand-waved.** "The model-usage audit already identified ~5 reclaimable H200s" is a forward-looking claim, not an allocation. Taking an H200 removes chat/embedding capacity, and the memory file records that a stray enabled per-GPU systemd unit on lynx silently stole a GPU and poisoned benchmarks. Also undefined: what happens to the queue when the sole video backend goes UNHEALTHY (currently: everything sits until `video_job_max_wall_seconds` and then fails in a batch). **Fix:** name the specific node and GPU index in Phase 0, verify no sibling unit is enabled on it, document the drain procedure (disable backend → runner stops claiming → in-flight completes), and define backend-down policy as "hold `queued`, surface 'service paused' in the UI, do not burn the wall clock" — i.e. the wall-clock deadline should only run while a backend is available.
